# Agent 岗面试复习资料

## 1. 项目一句话介绍

**基于 LangGraph 的校准证书文档核验 Agent 系统**。  
这是一个面向垂直业务场景的文档核验 Agent，而不是通用聊天机器人。系统围绕校准证书场景，完成 `PDF 解析 -> Markdown -> JSON -> 检索增强 -> 多维核验 -> 报告生成` 的端到端闭环，并进一步做了 FastAPI 服务化、Streamlit 前后端分层以及 Docker/Nginx 部署。

---

## 2. 面试时建议的项目定位

### 2.1 不要把项目讲成什么

- 不要讲成“我做了一个聊天机器人”
- 不要讲成“我只是调了大模型接口”
- 不要硬讲成“成熟的多智能体系统”

### 2.2 更合适的定位

- 业务决策型 Agent
- 工作流编排型 Agent
- Agent + RAG 的智能文档核验系统
- 规则引擎与 LLM 混合决策系统

### 2.3 一句话版本

> 我做的是一个面向校准证书场景的文档核验 Agent，把 PDF 解析、结构化抽取、多知识库 RAG、规则校验和 LLM 判定组织成一个可编排、可服务化、可部署的工作流系统。

---

## 3. 面试开场 1 分钟介绍

这个项目最初是一个脚本式的文档核验流程，我先完成了 PDF 解析、结构化抽取和规则核验的主链路，后续又把它迁移到 LangChain/LangGraph 架构，重构成一个可编排的工作流 Agent。系统的核心流程是先将校准证书从 PDF 转成 Markdown，再抽取成 JSON 结构化数据，随后结合多知识库 RAG、规则引擎和 LLM 推理，对完整性、环境条件、校准地点、建议周期、参数与不确定度等维度进行自动核验，最终生成报告。为了让系统更贴近真实工程落地，我又基于 FastAPI 做了服务化封装，让 Streamlit 前端通过 HTTP 调用后端接口，并进一步用 Docker Compose 和 Nginx 做了前后端分层部署。

## 5. 端到端主链路怎么讲

面试里推荐按下面顺序讲：

1. 用户上传校准证书 PDF
2. 系统先做 PDF 到 Markdown 的解析
3. 再把 Markdown 解析成结构化 JSON
4. 根据 JSON 内容触发多维核验模块
5. 在部分核验节点中接入多知识库 RAG，补充规则依据和上下文
6. 对参数、不确定度等复杂项采用“检索 + 规则 + LLM”的混合决策
7. 最后汇总各模块结果，生成核验报告
8. 通过 FastAPI 暴露任务提交、状态查询、报告获取接口
9. Streamlit 前端通过 HTTP 调用后端服务完成交互

这条主链路的关键词是：

- 中间表示设计
- 工作流编排
- 多知识库 RAG
- 混合决策
- 服务化
- 前后端分层

## 6. 各模块实现逻辑详解

## 6.1 `core/pipeline.py`：顺序版核心流程

这是整个项目最核心的业务编排文件，主要职责是把文档核验拆成清晰的阶段并顺序执行。

### 关键设计点

#### 1. `PipelineHooks`

这个类本质上是一个轻量级的回调机制，用来向外暴露：

- 状态更新
- 进度更新
- 普通日志
- 警告
- 错误
- 成功消息

它的价值在于让核心 pipeline 不依赖具体 UI 或 API。  
无论是 CLI、Streamlit 还是 FastAPI，都可以通过 hooks 感知流程状态。

#### 2. `json_cache_needs_refresh`

这个函数负责判断已有 JSON 缓存是否需要重建。  
核心思想是：如果已有解析结果足够新、输入没有变化，就尽量复用缓存，避免重复执行昂贵的文档解析和抽取流程。

这个设计体现的工程能力是：

- 缓存复用
- 减少重复计算
- 提升用户体验和运行效率

#### 3. `pdf_to_md_first_step`

这是 PDF 到 Markdown 的第一阶段转换，核心依赖 `pdf_md.py`，而 `pdf_md.py` 又依赖 `MinerU`。

这里做了几件事：

- 命中缓存时直接复用 Markdown
- 否则创建临时目录
- 调用文档解析模块生成 Markdown
- 检查 Markdown 文件是否真的生成
- 异常时通过 hooks 往外抛出错误信息

这一步的面试亮点是：

- 并不是直接把 PDF 丢给大模型
- 而是先构建一个对后续检索和推理更友好的中间表示

#### 4. `run_verification`

这是顺序版总入口。整体逻辑如下：

1. 初始化配置、输出目录、embedder 等运行时对象
2. 生成报告头部
3. 执行 PDF -> Markdown
4. 执行 Markdown -> JSON
5. 按顺序执行多个核验模块：
   - `InfoCheckRunner`
   - `EnvironmentCheckRunner`
   - `LocationCheckRunner`
   - `CycleCheckRunner`
   - `ParameterCheckRunner`
6. 如果完整性核验就已经判定失败，可以提前停止后续流程
7. 汇总结果并生成最终报告

### 这部分面试时怎么讲

> 我一开始先保留了一版顺序执行 pipeline，因为它更适合快速打通业务闭环。它的核心设计点是用 hooks 解耦核心流程和外部展示层，并通过缓存、异常处理和阶段化拆分，让文档核验链路具备可维护性和可观测性。后续在此基础上，我再把它迁移到 LangGraph 工作流版本。

  - 将环境变量应用到配置对象

### 面试价值

这个模块体现的是工程化意识。  
很多 demo 项目会把路径、模型名、输出目录写死在代码里，但这个项目把它抽成了统一配置模块，更适合服务化和部署。

## 6.3 `langchain_app/graph/state.py`：Agent 状态定义

这是 LangGraph 版本非常重要的一层。  
在 Agent 工作流里，状态对象决定了各节点之间如何传递信息。

### `VerificationState` 中保存了什么

- 输入路径
- Markdown / JSON 路径
- 配置与运行时对象
- embedder / llm client
- 各模块核验结果
- 日志、警告、错误
- 当前步骤
- 进度
- 是否提前停止
- 最终报告

### 为什么这个设计重要

因为 LangGraph 本质上是状态驱动工作流。  
每个节点不应该依赖全局变量，而应该只消费和更新状态对象。

### 面试时怎么讲

> 我在 LangGraph 版本里设计了统一的状态对象，把输入文件、解析中间结果、各核验模块输出、日志和错误都放进同一个状态结构里。这样每个节点都只需要读取和更新状态，就能保证工作流的清晰性和可扩展性。

---

## 6.4 `langchain_app/graph/verification_graph.py`：LangGraph 工作流

这是你讲“Agent 工作流编排”的关键文件。

### 节点拆分

工作流主要包含这些节点：

- `parse_pdf`
- `parse_json`
- `integrity_check`
- `environment_check`
- `location_check`
- `cycle_check`
- `parameter_check`
- `assemble_report`

### 核心亮点

#### 1. 节点化编排

原本串行脚本中的多个阶段，被重构成了语义清晰的独立节点。

#### 2. 状态流转

每个节点只关心输入状态和输出状态，便于扩展、调试和单测。

#### 3. 条件分支

完整性核验后，如果状态中的 `should_stop` 为真，就提前跳到报告汇总，而不是继续执行后续节点。

这就是比较典型的 Agent 工作流里的“条件路由”。

### 面试时怎么讲

> 我不是简单把 LangGraph 当成一个新框架去套，而是把原本脚本式的多阶段业务流程，重新抽象成状态驱动的节点图。这样系统具备了更好的扩展性，比如后续要新增一个核验节点，只需要加状态字段和图中的节点/边，而不需要重写整条主流程。

## 6.5 `langchain_app/checks/parameter/retrieval.py`：参数核验的检索层

参数核验是项目里最值得讲的模块之一，因为它体现了你对 RAG 的理解不是“只会向量检索”。

### 主要职责

- 查询知识库
- 将检索结果标准化
- 过滤低质量候选
- 按标准依据优先级选择更合适的知识项

### 关键函数

#### `search_calibration_data`

- 封装对 CNAS 检索服务的调用
- 根据 query 从向量库取回相关文档

#### `filter_kb_entries`

- 对召回结果进行业务规则过滤

#### `parse_kb_entry`

- 把知识库文档解析成更适合业务判断的结构

#### `select_best_kb_entries`

- 根据标准代码、内容匹配度等因素选择更优候选

### 面试时怎么讲

> 参数核验里的检索不是简单把 TopK 文本塞给模型，而是先做标准依据匹配、结构化解析和候选过滤，把更高质量、更贴近当前参数项的知识项筛出来，再进入后续判断环节。

## 6.6 `langchain_app/checks/parameter/semantic.py`：参数语义理解与候选筛选

这是项目中非常能体现深度的部分。

### 为什么它重要

单纯做向量检索有一个很常见的问题：  
召回结果可能语义相关，但未必业务上可用。

例如：

- 参数名称接近，但量纲不同
- 都是“误差”，但一个是示值误差，一个是重复性
- 都是“不确定度”，但业务含义不同

所以你在这里没有停留在“召回阶段”，而是进一步做了参数语义理解。

### 核心对象

- `ParamSemantic`
  - 表示当前参数项的语义特征
- `KbCapability`
  - 表示知识库条目对应的能力描述
- `SelectionAudit`
  - 记录候选选择过程中的审计信息
- `SelectionResult`
  - 输出最终选择结果

### 核心逻辑

#### `infer_param_semantics`

根据参数名称、上下文、正则和关键词，推断：

- 主物理量
- 单位族
- 条件轴
- 不确定度类型
- 任务意图

#### `infer_kb_capability`

把知识库条目转成统一能力表示，方便后续和参数语义做对齐。

#### `structured_prefilter`

先基于结构化语义做预筛选，而不是完全依赖 embedding 相似度。

#### `select_basis_with_audit`

最终做候选选择，并记录为什么选中、为什么过滤掉某些候选。

### 这部分是你项目里最容易打动面试官的点

因为它体现的是：

- 你知道 RAG 会有误召回
- 你知道业务场景不能只依赖 embedding
- 你通过“语义预筛选 + 审计信息”提升了可解释性和准确性

### 面试时怎么讲

> 我在参数核验里没有停留在简单的向量召回，而是增加了参数语义识别和候选预筛选机制。也就是说，我会先判断当前参数到底是在校验哪个物理量、哪类单位、哪类不确定度，再去匹配知识库能力项，这样能显著降低误召回进入最终判断的概率。

## 6.7 `api/app.py`：FastAPI 服务化入口

这是项目从“本地脚本”升级到“可交付系统”的关键一步。

### 主要职责

- 暴露统一 HTTP 接口
- 处理文件上传
- 创建任务
- 后台执行核验任务
- 提供状态查询与报告获取

### 当前主要接口

- `GET /api/v1/health`
  - 健康检查
- `POST /api/v1/tasks/verify`
  - 上传 PDF 并创建核验任务
- `GET /api/v1/tasks/{task_id}`
  - 查询任务状态、进度、日志
- `POST /api/v1/tasks/{task_id}/cancel`
  - 取消任务
- `GET /api/v1/tasks/{task_id}/report`
  - 获取最终报告

### 关键设计点

#### 1. 长任务任务化

文档核验可能耗时较长，所以不是同步阻塞请求，而是：

1. 提交任务
2. 返回 `task_id`
3. 前端轮询状态
4. 最后获取结果

这是典型的 AI 服务化模式。

#### 2. `dry_run`

为了方便验证服务层而不触发重型解析流程，设计了 `dry_run` 模式。

它的价值在于：

- 排查问题时先分离“服务层问题”还是“业务层问题”
- 快速验证接口、状态管理、报告返回逻辑

#### 3. 内存任务注册表

当前实现采用内存级任务管理，适合本地演示和单机服务。

### 面试时怎么讲

> 为了让项目更像真实业务系统，我把核心核验流程用 FastAPI 封装成了服务。由于文档核验是长任务，我没有直接做同步请求返回，而是设计成任务提交、状态轮询和结果获取的模式，这样更适合前端集成和后续扩展。

## 6.8 `app.py`：Streamlit 前端

这个文件原本是直接驱动业务逻辑的前端，现在已经改成了 API 客户端。

### 当前职责

- 上传文件
- 发起任务请求
- 调用健康检查
- 轮询任务状态
- 展示最终报告

### 为什么这次改造很重要

之前模式：

- Streamlit 直接调用 `run_verification()`

现在模式：

- Streamlit -> FastAPI -> 核心业务层

这意味着：

- 前端和后端完全解耦
- 后端能力可被其他前端复用
- 更接近生产环境系统架构

### 面试时怎么讲

> 早期为了快速验证，我让 Streamlit 直接驱动业务 pipeline；后续为了服务化和系统分层，我把 Streamlit 改造成 HTTP 客户端，只负责界面和交互，真正的业务逻辑统一收口到 FastAPI 后端。

## 8. 高频面试问题与回答思路

## 8.1 项目类问题

### Q1：先用一句话介绍一下你的项目

**参考回答：**

这是一个面向校准证书场景的文档核验 Agent 系统，我把 PDF 解析、结构化抽取、多知识库 RAG、规则校验和 LLM 推理组织成一个工作流，用于自动核验证书中的完整性、环境条件、校准地点、建议周期、参数和不确定度，并进一步做了 FastAPI 服务化和 Docker 部署。

---

### Q2：这个项目为什么叫 Agent，而不是普通 RAG 系统

**参考回答：**

因为它不只是“检索 + 生成”，而是一个多阶段任务工作流。系统会经历文档解析、结构化抽取、多维核验、条件路由、报告生成等多个步骤，并且不同阶段会调用不同能力模块和工具。尤其在 LangGraph 版本里，整个流程是基于状态对象和节点编排组织起来的，所以更准确地说是一个工作流型 Agent，而不是单纯的问答式 RAG。

---

### Q3：这个项目最难的点是什么

**参考回答：**

我认为最难的是参数核验。因为参数项名称、量纲、单位和不确定度表达方式都比较复杂，单纯依赖向量检索很容易误召回。所以我在这部分做了语义识别、候选预筛选、标准依据匹配、单位换算和工具约束，把它做成了“检索 + 规则 + LLM”的混合决策链路。

---

### Q4：为什么没有直接让大模型读 PDF

**参考回答：**

因为原始 PDF 噪声很大，版式、表格、换行、页眉页脚都会影响模型理解和后续检索稳定性。所以我先把 PDF 转成 Markdown，再抽取成 JSON 结构化数据，相当于设计了一层更适合下游推理的中间表示。这样做的好处是可控性更强，也更方便做规则校验和检索增强。

---

### Q5：你为什么要从脚本版迁移到 LangGraph

**参考回答：**

脚本版更适合快速打通业务闭环，但随着核验模块变多，流程会越来越长，扩展和调试成本也会上升。迁移到 LangGraph 后，我可以把流程拆成状态驱动的节点图，显式管理节点之间的数据流和条件分支，这样后续新增节点、替换模块或者做故障定位都会更方便。

---

## 8.2 RAG 类问题

### Q6：你项目里的 RAG 和普通知识库问答有什么不同

**参考回答：**

我这里的 RAG 是面向业务决策的，不是面向开放问答的。检索结果不是直接拿去回答用户问题，而是作为规则校验和模型推理的依据，参与最终的核验结论生成。所以重点不只是召回相关文本，而是要保证召回内容在业务上可用、可解释。

---

### Q7：为什么要做多知识库，而不是一个大向量库

**参考回答：**

因为不同核验维度对应的知识类型差异很大，比如 CNAS 标准、温度条件、校准地点和建议周期，它们的查询模式和使用场景都不同。拆成多知识库之后，每个核验节点可以按业务字段定向检索，能降低无关召回，提高检索质量和后续判断的稳定性。

---

### Q8：你是怎么优化 RAG 召回质量的

**参考回答：**

我主要做了三层优化。第一层是按业务维度拆知识库，降低检索空间。第二层是在参数核验里增加标准依据匹配和候选过滤，而不是盲目使用 TopK。第三层是做参数语义识别和结构化预筛选，让召回结果先经过业务约束，再进入最终判定。

---

### Q9：有没有做 rerank

**参考回答：**

这个项目里我没有用一个独立的通用 rerank 模型去做统一重排，而是更偏业务化地做了候选过滤和选择，比如标准代码优先、参数语义对齐、单位族匹配和结构化规则筛选。我的思路是，在这个场景里业务规则比通用文本重排往往更有效。

---

## 8.3 Agent / Workflow 类问题

### Q10：LangGraph 在你项目中解决了什么问题

**参考回答：**

它主要解决了复杂多阶段流程的可编排和状态管理问题。以前脚本版是顺序写死的，后续扩展和插拔模块不够自然；引入 LangGraph 后，我把各个阶段拆成节点，并把过程数据统一收进状态对象，流程就变成了显式可管理的状态图，也更适合做条件路由和可观测性扩展。

---

### Q11：你的 Agent 状态里放了什么，为什么这么设计

**参考回答：**

我把输入文件路径、Markdown 和 JSON 中间结果、运行时配置、embedder、各核验模块结果、日志、警告、错误、进度、当前步骤以及是否提前停止等信息都放进状态对象里。这样每个节点只处理当前状态并更新状态，不需要依赖全局变量，整个工作流会更清晰，也更方便调试。

---

### Q12：有没有做规划和工具调用

**参考回答：**

如果从广义上看，这个项目本身就是一种固定工作流规划，因为任务先被拆成文档解析、抽取、核验和报告生成几个阶段。工具调用方面，我把解析和核验能力做成独立模块和工具化封装，让不同节点按需调用。它不是开放式 agent planner，而是更偏业务工作流型的规划与执行。

---

## 8.4 工程化与服务化问题

### Q13：为什么要做 FastAPI 服务化

**参考回答：**

因为如果只停留在本地脚本或 Streamlit 直连核心逻辑，这个项目更像 demo。做成 FastAPI 以后，系统具备了标准化接口能力，可以被前端或其他系统调用，也更方便做任务状态管理、日志暴露和后续部署。

---

### Q14：为什么 Streamlit 还要改成通过 HTTP 调 FastAPI

**参考回答：**

这是为了前后端解耦。原来 Streamlit 直接调用核心逻辑，虽然开发快，但前端和业务耦合很深。改成通过 HTTP 调用后，Streamlit 就只是一个前端客户端，真正的业务都统一收口到 FastAPI 后端，这样系统结构更清晰，也更便于替换前端或做部署扩展。

---

### Q15：为什么文档核验接口要设计成“提交任务 + 查询状态”

**参考回答：**

因为文档核验属于长任务，直接同步阻塞请求不够稳，也不利于前端体验。所以我把它做成任务制，先提交任务拿到 task_id，再轮询状态，最后获取报告。这种设计在 AI 推理服务、Agent 服务里都比较常见。

---

### Q16：为什么还要做 Docker 和 Nginx

**参考回答：**

Docker 主要解决环境一致性和交付问题，让前后端服务都能容器化运行；Nginx 主要解决统一入口和生产风格部署问题，比如把 `/` 转给 Streamlit，把 `/api` 转给 FastAPI。这样整体架构更接近真实线上系统，而不是只在本地解释器里能跑。

---

## 8.5 测试与评估问题

### Q17：你怎么验证这个系统是有效的

**参考回答：**

我做了两类验证。第一类是样本评估，比如部分解析准确率达到 100%，参数核验综合准确率达到 95%。第二类是回归测试，包括 JSON 缓存逻辑、参数工具约束、检索一致性、FastAPI 服务层 smoke test 等。我的思路是既看业务效果，也看工程稳定性。

---

### Q18：这个项目有哪些测试是值得讲的

**参考回答：**

我会重点讲三类。第一类是 pipeline 和缓存相关测试，验证重复运行时是否正确复用中间结果。第二类是参数核验测试，验证单位、不确定度和工具约束逻辑。第三类是 FastAPI 服务层测试，验证健康检查、任务提交和 dry-run 报告链路，确保服务化改造后接口层是稳定的。

---

## 8.6 追问与反问类问题

### Q19：如果让你继续优化这个项目，你会怎么做

**参考回答：**

我会从三方面优化。第一，补强文档解析层的兜底能力，比如 MinerU 不可用时增加备用解析方案。第二，把当前内存任务注册表升级为更稳定的异步任务系统，比如 Redis + Celery 或消息队列。第三，补充更系统化的评测体系，包括分模块指标、召回分析和错误样本回流。

---

### Q20：这个项目目前最大的不足是什么

**参考回答：**

我觉得主要有三点。第一，它更像单 Agent 工作流，不是多 Agent 协作系统。第二，真实 PDF 解析依赖 MinerU，可移植性还可以继续增强。第三，目前任务状态管理仍偏单机内存实现，更适合演示和中小规模场景，若要更强生产化还需要引入持久化任务系统和监控能力。

## 12. 常见追问的回答技巧

### 技巧 1：少说“我用了什么框架”，多说“它解决了什么问题”

不要只说：

- 我用了 LangGraph
- 我用了 FastAPI
- 我用了 Chroma

更好的表达：

- 我用 LangGraph 解决多阶段工作流的状态管理和条件路由问题
- 我用 FastAPI 完成服务化和任务状态接口封装
- 我用 Chroma 支撑不同业务维度的知识检索

### 技巧 2：少说“准确率很高”，多说“我是怎么提升的”

不要只背指标。  
更重要的是说清楚：

- 误差来自哪里
- 你做了什么优化
- 为什么这些优化有效

### 技巧 3：坦诚边界，比硬吹更有说服力

可以明确说：

- 这是单 Agent 工作流，不是复杂多智能体
- 当前任务系统更适合单机或演示场景
- MinerU 是关键依赖，容器化时要考虑兼容问题

这不会减分，反而说明你理解系统边界。
