# 参数核验模块需求说明

本文根据 `需求文档/构建智能化核验系统-0410.docx` 和当前 `langchain_app/` 主线代码整理参数核验模块的需求边界，回答三个核心问题：

- 如何比较？
- 和谁比较？
- 判定规则是什么？

## 1. 模块目标

参数核验模块用于判断证书中的参数表项目是否满足对应校准规范或能力库要求。

它不是做“证书参数行之间的互相比较”，而是做：

```text
证书参数行
  vs
同校准依据 / 同能力项 / 同语义类型的 KB 能力库候选记录
```

当前主流程以 `langchain_app/checks/parameter/parameter.py` 为入口，核心链路是：

```text
证书 JSON 参数行
-> 参数合同化 ParameterContractV2
-> 证书参数语义识别
-> 按仪器 rule_set 限定允许的 semantic target
-> RAG / KB 检索同依据候选
-> selector 选择最匹配 KB 候选
-> validator 执行范围、误差、不确定度比较
-> 汇总 PASS / FAIL / REVIEW / ERROR
-> Markdown 报告
```

## 1.1 需求文档中的业务规则

需求文档对“校准项目、范围及不确定度”和“每个测量点结论”提出了更完整的人工核验口径：

- 校准项目、范围必须涵盖被测仪器实际使用范围及关键性能指标。
- 校准项目、范围和不确定度必须在《认可的校准和测量能力表》规定内容内。
- 如果校准项目或范围超出能力表规定，证书应强制标注 `▽` 标识；未标识时视为不符合要求。
- 测量结果的不确定度如果优于能力表规定内容，视为不符合要求。
- 每个测量点如有技术指标，应按 JJF 1094-2002《测量仪器特性评定》的符合性判定逻辑检查证书内页结论。
- 当示值误差的不确定度与最大允许误差绝对值之比 `<= 1:3` 时，示值误差在最大允许误差限内可判为 `P`，超出则为 `F`。
- 当上述比值 `> 1:3` 时：
  - `|示值误差| <= |最大允许误差| - 扩展不确定度` 可判为 `P`。
  - `|示值误差| >= |最大允许误差| + 扩展不确定度` 可判为 `F`。
  - 介于两者之间时，按中心要求可判为 `P*`。

当前代码已经覆盖“项目/范围/误差/不确定度是否落在能力库内”的主要自动核验，但 `▽` 标识核验、JJF 1094 的 `P/F/P*` 结论复核、标准器 1:3 精度比核验还没有形成完整独立规则，应作为后续迭代验收项。

## 2. 输入与输出

### 2.1 输入

参数核验至少依赖三类输入。

第一类是 PDF 解析模块输出的证书 JSON：

```json
{
  "仪器名称": "频率计数器",
  "型号规格": "...",
  "校准依据": "JJG ...",
  "依据参数_中间数据": [
    {
      "项目名称": "频率准确度",
      "数据明细": {},
      "__parameter_contract": {},
      "__normalized_fields": {},
      "__parser_meta": {}
    }
  ]
}
```

参数表行建议额外保留：

- `result_flag` / `结论`：证书内页的 `P`、`F`、`P*` 等测量点结论。
- `limit_value` / `允许误差`：被测仪器最大允许误差或技术指标限值。
- `out_of_scope_mark` / `▽` 标识：项目或范围超出认可能力表时的证书标记。
- 原始表头和原始单元格文本：用于追溯单位继承、乱码、跨页表格和人工复核。

第二类是 RAG / KB 检索返回的候选能力库记录：

```json
{
  "file_code": "JJG ...",
  "measured": "频率准确度",
  "measure_range_text": "10 Hz ~ 18 GHz",
  "error_limit_text": "±1e-7",
  "uncertainty": "U=...",
  "scope_mark_required": false,
  "source": "能力库来源"
}
```

第三类是仪器规则上下文，由 `rule_sets/router.py` 根据仪器名称、校准依据、PDF 分类解析：

```json
{
  "profile_id": "time_frequency.counter",
  "family": "time_frequency",
  "semantic_whitelist": [
    "frequency_accuracy",
    "period_accuracy",
    "input_sensitivity",
    "reference_oscillator"
  ]
}
```

### 2.2 输出

每条参数行输出一条核验记录，最终汇总为 Markdown 报告。单条记录至少需要包含：

```json
{
  "status": "PASS / FAIL / REVIEW / ERROR",
  "project_name": "频率准确度",
  "basis_code": "JJG ...",
  "semantic_target": "frequency_accuracy",
  "selected_kb": {
    "measured": "频率准确度",
    "measure_range_text": "...",
    "error_limit_text": "...",
    "uncertainty": "..."
  },
  "range_result": {},
  "error_result": {},
  "u_result": {},
  "reason": "范围、误差、不确定度的综合说明"
}
```

## 3. 和谁比较

参数核验的比较对象不是任意 KB 结果，而是经过 selector 过滤和排序后的候选。

### 3.1 第一层：同依据

证书参数行会提取 `basis_code`，KB 候选也会提取 `file_code`。优先要求二者规范化后相同。

```text
证书 basis_code == KB file_code
```

如果已经存在同依据候选，但单条参数没有同能力项候选，不能直接判定证书 FAIL，应进入 REVIEW，原因是“KB 无对应参数，需人工核验”。

这里要区分两个层级：

- 整个依据层面：如果向量库访问失败、证书依据无法解析出 JJG/JJF/GJB 规程代号，或 KB 中完全没有同规程条目，当前主流程会把该依据下的参数行按 ERROR 记录。
- 单条参数层面：如果已经存在同规程 KB 条目，但某个证书参数行找不到同能力项 / 同语义候选，当前代码把该行判为 REVIEW。

### 3.2 第二层：同语义能力项

每条证书参数行会被识别为一个 `semantic_target`，例如：

- `frequency_accuracy`: 频率准确度
- `frequency_range`: 频率范围
- `period_accuracy`: 周期 / 时间间隔准确度
- `reference_oscillator`: 参考振荡器 / 频标类参数
- `input_sensitivity`: 输入灵敏度
- `power_accuracy`: 功率准确度
- `phase_noise`: 相位噪声
- `modulation_quality`: 调制质量
- `dynamic_range`: 动态范围

KB 候选也会被规范化成能力项。只有语义目标兼容的候选才能进入比较。

### 3.3 第三层：仪器规则白名单

当前已经引入 `rule_sets/`，目的不是简单加一个 profile 门禁，而是让不同仪器走不同参数规则。

例如：

- 时间频率计数器允许频率、周期、计数、输入灵敏度、参考振荡器相关规则。
- 射频微波仪器允许功率、相位噪声、调制质量、频谱纯度、动态范围等规则。
- 示波器 / 函数发生器暂时只启用已建模的频率、周期、功率、动态范围规则。
- 未识别仪器走 `generic.default`。当前 generic 规则包会启用通用 semantic targets，不会自动把所有结果改成 REVIEW；只有语义未知、字段不足、候选不兼容或 KB 缺项时才进入 REVIEW。

如果某个参数语义不在当前仪器规则包允许范围内，当前代码会把它降为 `unknown`，进入人工复核路径，避免误套其他仪器的规则。

### 3.4 第四层：单位、子类型、条件轴

selector 还会继续检查：

- 单位族是否兼容，例如频率、时间、功率、电压、比值。
- semantic subtype 是否一致，例如日差、周期、频率稳定度、动态范围等。
- 条件轴是否匹配，例如频点、时间间隔、功率电平、调制条件。
- 参考量是否匹配，例如 `reference_value` / `measure_value` 是否应作为比较基准。

最终选择排序最高的 `selected_candidate` 作为本条参数的 KB 对照。

## 4. 如何比较

当前实际比较集中在 `validator.py` 的三类函数：

- `verify_range_logic()`
- `verify_error_logic()`
- `verify_uncertainty_logic()`

调度位置在 `parameter.py::_evaluate_selected_kb_results()`。

### 4.1 范围比较

比较对象：

```text
证书项目 / 测量点 / 条件轴 / 参考点
  vs
KB measured / measure_range_text / selected_candidate 的规范化范围
```

常见规则：

- 证书项目应能匹配 KB 能力项；匹配不到同能力项时进入 REVIEW。
- KB 是上下限范围：证书测量点在 `[lower, upper]` 内则 PASS，否则 FAIL。
- KB 是单边限制：按 `<`、`<=`、`>`、`>=` 比较。
- KB 是对称范围或 `±` 限制：取绝对值后比较。
- KB 是离散频点列表：证书频点必须命中列表中的点，允许极小数值容差。
- KB 范围无法解析但包含数字时，当前代码可能保守 PASS 并保留文本说明；完全无法解析时 ERROR。
- 测量值或范围缺失时，当前代码返回 PASS/Skip，报告层应把“缺数据跳过”作为风险点展示。

特殊规则：

- 输入灵敏度有两套特殊逻辑：一是业务 override，只检查文本是否存在乱码；文本正常可直接 PASS，乱码或异常文本直接 FAIL，并跳过依据核验。二是复合范围判断，同时检查频率轴和电平 / 灵敏度幅度范围；复合子项出现 FAIL 或 ERROR 时整体按 FAIL。
- 参考振荡器类参数如果探针来源不明确，即使范围比较可通过，也可能被改为 REVIEW。
- 部分“证书能力优于 KB 下边界”的情况会进入 REVIEW，而不是直接 FAIL。
- 需求文档要求：超出能力表规定的项目或范围必须有 `▽` 标识；当前代码尚未完整实现该标识核验。

### 4.2 误差比较

比较对象：

```text
证书 error_value / 示值误差 / 频率偏差
  vs
KB error_limit_text / limit_value
```

常见规则：

- KB 是绝对误差限：比较 `abs(证书误差) <= abs(KB 限值)`。
- KB 是百分比限值：将证书误差换算为相对误差后比较。
- KB 是单边限制：按 `<`、`<=`、`>`、`>=` 比较。
- KB 是范围限制：证书误差落在 `[lower, upper]` 内则 PASS，否则 FAIL。
- KB 是对称范围或 `±` 限值：取证书误差绝对值比较。

需要 REVIEW 的典型情况：

- KB 是百分比限值，但证书测量值缺失或为 0，无法把绝对误差换算成相对误差。
- 证书字段看起来是误差，但单位或上下文无法确认比较基准。

当前代码中，如果误差值格式无法解析，部分路径会默认 PASS/Skip。这是兼容旧逻辑的行为，后续建议改成 REVIEW，避免解析失败被误认为合格。

另外，误差值或限值任一缺失时，当前 `verify_error_logic()` 返回 PASS/Skip。这是现状，不代表业务上已经确认合格，报告层应把它作为字段缺失风险。

### 4.3 不确定度比较

比较对象：

```text
证书 cert_u
  vs
KB uncertainty / kb_u
```

常见规则：

- 证书不确定度和 KB 不确定度都是绝对值：当前代码按 `证书 U >= KB_U` 判 PASS，否则 FAIL。
- 二者都是相对值：按相对系数直接比较，仍然是 `证书 Urel >= KB_Urel` 判 PASS。
- 一边是相对值、一边是绝对值：用测量值换算到同一量纲后比较。
- 如果任一方是不确定度区间：当前代码按区间边界比较；证书区间整体不低于 KB 区间或二者存在重叠时 PASS，证书区间整体低于 KB 区间时 FAIL。

这里的 KB_U 不能按“最大允许不确定度上限”理解。当前实现和测试都把它作为能力库 / CMC 的可报能力下限：证书报告的不确定度不能小于机构在能力库中声明的能力。也就是说，误差限是“不能超过”，但不确定度能力项是“不能低报”。

需要 REVIEW 的典型情况：

- 相对不确定度需要测量值换算，但测量值缺失或为 0。
- 证书 U 和 KB U 的表达方式不一致，无法可靠确认覆盖因子、置信概率或单位。
- 某些跨目标 fallback 策略下，不确定度比较会按策略跳过，并把风险写入原因。

不确定度的缺失和解析失败也要区分：

- 测量值、证书 U 或 KB_U 任一缺失时，当前 `verify_uncertainty_logic()` 返回 PASS/Skip。
- 测量值无法解析，或证书 U / KB_U 存在但无法解析时，当前 validator 返回 ERROR。

### 4.4 测量点结论比较

比较对象：

```text
证书内页 result_flag / 结论
  vs
按误差、最大允许误差、不确定度计算出的应有结论
```

需求规则来自 JJF 1094-2002：

- 如果 `U / |MPE| <= 1/3`：
  - `|error| <= |MPE|`，应为 `P`。
  - `|error| > |MPE|`，应为 `F`。
- 如果 `U / |MPE| > 1/3`：
  - `|error| <= |MPE| - U`，应为 `P`。
  - `|error| >= |MPE| + U`，应为 `F`。
  - 其余灰区按中心要求可为 `P*`。

当前代码已在解析合同中保留 `result_flag` 相关字段，但参数核验主流程尚未把 `result_flag` 作为独立判定项与 JJF 1094 计算结论进行比对。后续建议新增 `verify_result_flag_logic()`，并把结果作为第四个子项并入综合状态。

## 5. 综合判定规则

当前选中 KB 候选后的综合状态规则在 `parameter.py::_resolve_selected_kb_status()`：

```text
任一子项为 FAIL
=> overall FAIL

没有 FAIL，但范围 / 误差 / 不确定度任一子项为 REVIEW
=> overall REVIEW

没有 FAIL / REVIEW，但检测到源数据异常或语义歧义
=> overall REVIEW

否则
=> overall PASS
```

注意：当前综合函数没有单独把子项 `ERROR` 提升为 overall `ERROR`，如果 validator 返回 ERROR 但没有 FAIL/REVIEW，理论上可能落到 PASS。后续建议把 `ERROR` 明确提升为 `REVIEW` 或 `ERROR`，这是一个需要修正的验收点。

同一文件的输入灵敏度复合范围函数例外：复合范围子项中如果出现 ERROR，会被合并为 FAIL。

需求完整落地后，综合状态应从三项扩展为四项：

```text
范围 result
误差 result
不确定度 result
测量点结论 result_flag result
```

其中 `▽` 标识核验可以作为范围覆盖核验的补充子项，或作为单独的 scope_mark 子项。

## 6. PASS / FAIL / REVIEW 口径

### PASS

满足以下条件时可以 PASS：

- 找到同依据、同语义能力项、单位和条件轴兼容的 KB 候选。
- 范围、误差、不确定度均满足 KB 要求。
- 没有源数据异常、语义歧义或策略跳过风险。

### FAIL

满足以下条件时可以 FAIL：

- 已选中明确可比较的 KB 候选。
- 证书测量点超出 KB 范围。
- 证书误差超过 KB 允许误差。
- 证书不确定度低于 KB_U，即低于能力库 / CMC 声明的可报能力下限。
- 项目或范围超出能力表规定且未按需求文档标注 `▽`。
- 证书内页 `P/F/P*` 结论与按 JJF 1094 计算出的应有结论明确不一致。

FAIL 必须建立在“对照对象明确且字段可比较”的基础上。

### REVIEW

以下情况应进入 REVIEW：

- 已经存在同依据 KB 条目，但单条参数找不到同能力项 KB 候选。
- 证书参数语义无法识别。
- 参数语义不在当前仪器规则包白名单内。
- 单位、条件轴、子类型、参考量不明确。
- 百分比 / 相对量需要换算但缺少测量值。
- 参考振荡器、输入灵敏度、GNSS 特殊参数等存在上下文歧义。
- RAG 检索结果不足以支撑自动判定。

### ERROR

ERROR 应用于系统异常或不可恢复解析错误，例如：

- 向量库不可用。
- KB 文档结构损坏。
- JSON 结构严重不符合契约。
- validator 抛出异常。
- 证书依据无法解析出规程代号，或 KB 中完全没有同规程条目。

业务上的“同依据下某个参数行没有匹配能力项”不应直接归为 ERROR，更不应归为 FAIL；当前代码按 REVIEW 处理。

## 8. 后续需求建议

### 8.1 把 ERROR 合并规则补齐

建议调整 `_resolve_selected_kb_status()`：

```text
FAIL 优先级最高
ERROR 次之
REVIEW 再次
PASS 最后
```

或者保守地把子项 ERROR 合并为 REVIEW，避免解析失败被误判为 PASS。

### 8.2 把参数核验继续抽成清晰子模块

当前 `parameter.py` 仍然承担了太多职责，建议后续拆为：

```text
checks/parameter/
  pipeline.py       # 单证书 / 批处理调度
  row_context.py    # 单条参数行上下文构造
  kb_matching.py    # KB 检索、候选解析、selector 调用
  evaluation.py     # range/error/u 三项比较与综合判定
  result_flag.py    # JJF 1094 P/F/P* 结论复核
  scope_mark.py     # 超能力范围 ▽ 标识核验
  report.py         # Markdown 渲染
```

这样不同成员可以分别维护：

- PDF 解析组：保证参数 JSON 契约。
- RAG 组：保证 KB 候选字段完整。
- 参数核验组：维护仪器规则包、比较器和报告。

### 8.3 每类仪器补充独立 fixture

每个仪器规则包至少需要三类样例：

- PASS：明确满足 KB 要求。
- FAIL：明确超出 KB 要求。
- REVIEW：字段缺失、语义不明或 KB 缺项。

建议 fixture 放在：

```text
tests/fixtures/parameter/
  time_frequency/
  rf_microwave/
  scope_function/
  generic/
```

### 8.4 明确 KB 字段质量要求

RAG / KB 组需要保证参数核验可直接使用以下字段：

- `file_code`
- `measured`
- `measure_range_text`
- `error_limit_text`
- `uncertainty`
- `scope_mark_required` / 超范围标识规则

- `source`

如果字段缺失，应返回诊断信息，而不是让参数核验模块靠文本猜测。

### 8.5 补齐需求文档规则

后续建议按优先级补齐：

1. 新增 `▽` 标识核验：当项目或范围超出 KB 能力范围时，检查证书是否标注 `▽`，未标注则 FAIL，已标注则 REVIEW 或按超范围说明处理。
