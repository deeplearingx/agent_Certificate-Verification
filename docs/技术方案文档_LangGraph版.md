# 智能文档核验系统 - 技术方案文档（LangGraph架构版）

## 一、项目概述

本项目是基于 LangGraph 编排框架重构的 AI 智能文档核验系统，完全保留原始 LlamaIndex 版本的功能，并提供更强的流程编排能力、更好的调试跟踪和可扩展性。系统结合了 OCR 技术、大语言模型（LLM）和向量数据库检索，能够对 CNAS 校准证书进行完整性、准确性和合规性检查。

**架构优势**：
- 统一编排：使用 LangGraph 作为主流程编排层
- 简化架构：核心组件更少，逻辑更清晰
- 强大生态：访问 LangChain 的大量工具和集成
- 更好调试：使用 LangSmith 进行追踪和调试
- 高度扩展：支持动态工具绑定和 Agent 行为调整
- 功能完整：完全保留原始项目的所有功能

## 三、技术方案详解

### 3.1 整体技术架构

系统采用 LangGraph 编排框架 + LangChain 能力层的架构，从上到下依次为：

1. **前端展示层**：Streamlit Web界面，提供用户交互
2. **业务编排层**：LangGraph状态图，统一流程管理
3. **节点执行层**：各功能节点，负责具体执行任务
4. **能力服务层**：LangChain工具库，提供核心能力
5. **数据检索层**：Chroma DB向量数据库 + bge-m3嵌入模型
6. **AI模型层**：DeepSeek LLM API + 本地bge-m3嵌入模型

**项目架构图**：
```
langchain_app/
├── app.py                      # Streamlit 主应用
├── core/                       # 核心功能
│   ├── llm_client.py           # LLM 客户端

│   └── pipeline.py            # Graph 入口兼容层

│   └── parsing.py             # PDF 解析服务
├── graph/                      # LangGraph 流程编排
│   ├── state.py               # 状态定义

    └── config.py               # 配置管理
```

### 3.2 各流程技术实现详解

#### 流程一：PDF → MD 转换（parse_pdf 节点）

**技术手段**：
- **OCR引擎**：MinerU混合引擎
- **解析方式**：PDF文本提取 + OCR识别双重方式
- **语言支持**：中文模式

**如何开发**：
1. 节点函数 `parse_pdf_node` 接受状态对象
2. 调用 `pdf_to_md_first_step()` 函数执行解析
3. 设置进度（0.1 → 0.2）并记录日志
4. 处理解析结果：成功则保存MD路径，失败则添加错误
5. 支持提前终止机制（should_stop）

**与需求结合**：
- 这是系统的入口流程，为后续所有核验环节提供数据基础
- 准确的PDF解析是保证后续核验准确性的前提

---

#### 流程二：MD → JSON 解析（parse_json 节点）

**技术手段**：
- **解析器**：规则引擎（无LLM依赖）
- **解析方式**：基于正则表达式和规则匹配
- **输出格式**：结构化JSON

**如何开发**：
1. 节点函数 `parse_json_node` 接受状态对象
2. 调用 `parse_md_to_json()` 函数解析MD文件
3. 提取两部分信息：
   - Meta元数据：仪器名称、型号、制造厂、证书编号、日期、环境信息等
   - Rows参数表：每个校准项目的参考值、示值、误差、不确定度
4. 记录解析过程的日志和进度

**与需求结合**：
- 提取的Meta元数据服务于"证书类别"、"被测仪器信息完整性"、"环境条件"、"校准地点"、"日期选择"等需求
- 提取的Rows参数表服务于"校准项目、范围及不确定度合规性"等需求

---

#### 流程三：信息完整性核验（integrity_check 节点）

**技术手段**：
- **规则校验**：字段完整性检查
- **LLM增强**：DeepSeek API进行语义合理性核验
- **CNAS阻断**：非CNAS证书直接终止核验

**如何开发**：
1. 节点函数 `integrity_check_node` 接受状态对象
2. 调用 `check_certificate_integrity()` 函数
3. 执行规则检查：验证必填字段是否存在
4. 执行CNAS阻断：检查是否为CNAS证书
5. 执行LLM语义核验：
   - 检查字段是否有乱码、空值
   - 验证仪器名称、型号、制造商逻辑一致性
   - 重点关注温度和相对湿度的合理性
6. 处理结果：成功继续流程，失败设置should_stop=True

**与需求结合**：
- 直接对应需求："证书类别"、"被测仪器信息完整性"、"模板选择合规性"
- 间接支撑："结论选择合规性"的综合判定

---

#### 流程四：环境条件核验（environment_check 节点）

**技术手段**：
- **向量检索**：Chroma DB检索温度要求数据库
- **语义匹配**：bge-m3嵌入模型计算相似度
- **LLM判断**：DeepSeek判断环境条件是否符合要求

**如何开发**：
1. 节点函数 `environment_check_node` 接受状态对象
2. 调用 `check_environment()` 函数
3. 从JSON中提取当前温度、相对湿度、校准依据
4. 使用校准依据作为查询，在温度要求向量数据库中检索TopK条相关记录
5. 构建上下文：检索结果 + 当前环境条件
6. 调用LLM进行判断并输出详细对比说明

**与需求结合**：
- 直接对应需求："环境条件合理性"

---

#### 流程五：校准地点核验（location_check 节点）

**技术手段**：
- **向量检索**：Chroma DB检索认可的校准地点
- **双阈值匹配**：必须匹配阈值 + 可选匹配阈值
- **LLM增强**：判断地点具体性和合理性

**如何开发**：
1. 节点函数 `location_check_node` 接受状态对象
2. 调用 `check_location()` 函数
3. 从JSON中提取校准地点数组
4. 使用地点向量进行检索，获取相似度最高的认可地点
5. 应用双阈值判断策略
6. （可选）调用LLM判断地点描述的具体性和合理性

**与需求结合**：
- 直接对应需求："校准地点合理性"

---

#### 流程六：校准周期核验（cycle_check 节点）

**技术手段**：
- **双数据库查询**：华为数据库 + 通用周期数据库
- **默认规则**：默认校准周期12个月
- **日期逻辑**：检查校准日期、签发日期合理性

**如何开发**：
1. 节点函数 `cycle_check_node` 接受状态对象
2. 调用 `check_cycle()` 函数
3. 从JSON中提取：建议校准周期、仪器名称、校准日期、签发日期
4. 优先查询华为周期数据库（针对华为设备）
5. 后备查询通用周期数据库
6. 兜底规则：若两库均无匹配，默认12个月
7. 日期逻辑校验：验证校准日期 ≤ 签发日期，检查时间差是否合理

**与需求结合**：
- 直接对应需求："日期选择合理性"

---

#### 流程七：参数与不确定度核验（parameter_check 节点）

**技术手段**：
- **向量检索**：Chroma DB检索CNAS校准能力数据库
- **语义理解**：推断参数量纲、匹配校准项目
- **单位换算**：自动换算单位，统一量纲
- **不确定度分析**：检查不确定度合理性
- **并发处理**：并行处理多个校准点

**如何开发**：
1. 节点函数 `parameter_check_node` 接受状态对象
2. 调用 `check_parameter()` 函数
3. 解析JSON参数表，遍历每个校准项目
4. 对每个校准项目执行以下步骤：
   - 语义推断：理解参数是什么（量纲、类型）
   - 向量检索：用项目名称和仪器信息检索CNAS数据库
   - 依据选择：从检索结果中选择最匹配的校准规范
   - 单位解析：提取数值和单位，支持科学计数法
   - 单位换算：统一到标准单位，支持倍数前缀
   - 范围检查：验证测量值是否在CNAS认可的测量范围内
   - 不确定度检查：验证证书给出的不确定度是否合理
5. 并发加速：使用线程池并行处理多个校准点

**与需求结合**：
- 直接对应需求："依据选择合规性"、"校准项目、范围及不确定度合规性"、"标准器选择合理性"、"每个校准点结论合规性"
- 间接支撑："结论选择合规性"的综合判定

---

#### 流程八：报告生成与输出（assemble_report 节点）

**技术手段**：
- **Markdown格式**：易读易编辑的报告格式
- **模板渲染**：模块化报告拼接
- **结果汇总**：整合各模块核验结果

**如何开发**：
1. 节点函数 `assemble_report_node` 接受状态对象
2. 调用 `VerificationReport` 类
3. 收集各核验模块的报告
4. 按照标准格式拼接成完整报告：
   - 添加报告标题和配置信息
   - 添加完整性核验报告
   - 添加环境条件核验报告（带分隔线）
   - 添加校准地点核验报告（带分隔线）
   - 添加校准周期核验报告（带分隔线）
   - 添加参数与不确定度核验报告（带分隔线）
5. 总结整体核验结论：通过/不通过
6. 保存为Markdown文件

**与需求结合**：
- 直接对应需求："核验签字"、"签发环节"

### 3.3 核心技术特点（LangGraph架构）

#### 1. 统一状态管理

**技术手段**：
- **Pydantic模型**：VerificationState继承自BaseModel
- **不可变数据**：状态通过节点传递，各节点只修改自己负责的部分
- **流程控制**：should_stop字段控制流程是否提前终止
- **完整追踪**：包含日志、警告、错误信息

**如何开发**：
```
状态对象包含：
- 文件路径：source_pdf_path, md_path, json_path
- 配置与运行时：config, runtime_cfg, embedder, llm_client
- 核验结果：integrity_result, environment_result, location_result, cycle_result, parameter_result
- 报告与日志：final_report, logs, warnings, errors
- 流程控制：should_stop, current_step, progress
```

---

#### 2. 条件流程编排

**技术手段**：
- **LangGraph条件边**：根据状态值动态选择下一个节点
- **should_stop机制**：支持提前终止流程
- **串行流程**：大部分节点按顺序执行
- **报告组装分支**：完整性核验失败直接跳转到报告组装

**如何开发**：
```
graph.add_conditional_edges(
    "integrity_check",
    lambda state: "assemble_report" if state.should_stop else "environment_check",
    {
        "assemble_report": "assemble_report",
        "environment_check": "environment_check",
    },
)
```

---

#### 3. 完整兼容性

**技术手段**：
- **API完全兼容**：与原始LlamaIndex版本接口一致
- **向量数据库格式不变**：Chroma DB数据无需转换
- **缓存系统兼容**：local_pdf、local_md、local_json缓存无需改动
- **配置接口兼容**：AppConfig类保持不变

**如何开发**：
```python
# 原始代码
from config.settings import get_app_config
from core.pipeline import run_verification

# LangGraph版代码
from langchain_app.utils import get_app_config

```

## 四、项目架构优势

### 4.1 架构对比

| 特性 | LlamaIndex版本 | LangGraph架构版 |
|-----|--------------|---------------|
| **学习曲线** | 较陡，LlamaIndex概念多 | 平缓，LangChain生态更熟悉 |
| **流程可视化** | 困难，分散在代码中 | 容易，LangSmith可视化执行 |
| **调试难度** | 高，难以追踪流程 | 低，每个节点的输入输出详细记录 |
| **扩展性** | 复杂，需要理解LlamaIndex架构 | 简单，新增节点只需继承模式 |
| **工具集成** | 受限，LlamaIndex工具系统 | 强大，LangChain工具库丰富 |
| **Agent集成** | 复杂，需要学习LlamaIndex Agent | 简单，标准LangChain Agent接口 |
