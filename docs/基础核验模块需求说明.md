# 基础核验模块需求说明

本文整理环境条件、校准地点、校准周期三类基础核验需求。需求来源包括：

- `需求文档/构建智能化核验系统-0410.docx`
- 当前 LangChain 主线代码：`langchain_app/checks/environment.py`、`location.py`、`cycle.py`
- 对应 RAG 检索模块：`langchain_app/retrieval/temperature.py`、`address.py`、`cycle.py`、`cnas.py`

## 1. 总体目标

基础核验模块负责在参数核验之前完成证书基础合规检查：

```text
环境条件核验
-> 校准地点核验
-> 校准周期与日期逻辑核验
```

这三类核验都依赖 PDF 解析后的 JSON 字段，并部分依赖向量库或 LLM 辅助判断。它们的结果应能被最终报告直接引用，并能明确区分：

- `PASS`：字段完整，且满足规则或知识库要求。
- `FAIL`：字段明确违反业务规则。
- `REVIEW`：字段缺失、知识库缺项、语义不确定，需要人工复核。
- `ERROR`：系统、配置、向量库、JSON 结构或检索链路异常。

当前代码中部分模块仍使用 `[完成]`、`[错误]`、`[警告]`、`待定` 等文本标签，后续建议统一映射为 `PASS / FAIL / REVIEW / ERROR`。

## 2. 环境条件核验

### 2.1 业务需求

需求文档中，环境条件包含温度和相对湿度，记录方式允许：

- 单值记录，例如 `23.5℃`、`50%`
- 区间记录，例如 `22.0℃～22.3℃`、`50%～55%`

核验要求：

- 温度、相对湿度的记录格式应规范合理。
- 温度、相对湿度应符合所依据校准规范、检定规程规定的环境条件要求。
- 如果证书内页也记录了环境参数，内页记录应与证书声明内容一致。
- 环境条件超出规定要求时，应视为技术性偏离。
- 格式错误或超限应进入异常处理。

### 2.2 输入

```json
{
  "仪器名称": "string",
  "校准依据": ["JJF/JJG/GJB ..."],
  "温度": "23.5℃ 或 22.0℃～22.3℃",
  "相对湿度": "50% 或 50%～55%",
  "温度_内页": "可选",
  "相对湿度_内页": "可选"
}
```

RAG / 温湿度库建议返回：

```json
{
  "仪器名称": "string",
  "FILE_CODE": "JJF/JJG/GJB ...",
  "FILE_NAME": "依据名称",
  "温度要求": "20℃～26℃",
  "相对湿度要求": "≤80%",
  "最大温度变化范围": "≤1℃/h",
  "认可组织": "广州实验室",
  "source": "来源",
  "distance": 0.0
}
```

### 2.3 如何比较

环境核验需要做两类比较。

第一类是证书环境值与知识库要求比较：

```text
证书温度 / 相对湿度
  vs
温湿度知识库中的温度要求 / 湿度要求 / 最大温差
```

比较规则：

- 单值温度应落在 KB 温度范围内。
- 区间温度的上下界应整体落在 KB 温度范围内。
- 湿度单值或区间应满足 KB 湿度上限或范围要求。
- 如果 KB 给出最大温差，证书记录为区间时应检查区间差值是否超出最大温差。
- 缺少 KB 要求时不应直接 FAIL，应进入 REVIEW。
- 向量库访问异常应进入 ERROR。

第二类是首页与内页一致性比较：

```text
温度 == 温度_内页
相对湿度 == 相对湿度_内页
```

比较前需要做格式归一化，例如：

- `°C` 与 `℃` 统一。
- `～`、`~`、`-` 表示的范围统一。
- 全角百分号统一为 `%`。

### 2.4 当前实现

当前主代码位置：

- `langchain_app/checks/environment.py`
- `langchain_app/retrieval/temperature.py`
- `langchain_app/graph/nodes/environment_check.py`

当前实现行为：

- 从 JSON 读取 `仪器名称`、`温度`、`相对湿度`、`校准依据`。
- 使用 `extract_numbers_from_str()` 提取温湿度数值，但当前只取第一个数值作为当前温度 / 湿度。
- 调用 `TemperatureRetrievalService.search_temperature_requirements()` 检索温湿度库。
- 检索 query 当前为 `"{criterion} 广州实验室"`，并过滤 `认可组织 == "广州实验室"`。
- 检索到有效记录后，调用 LLM 判断温度、湿度、温差是否符合要求。
- 如果无有效温湿度库记录，当前报告输出“待定”表格，而不是统一的 REVIEW。

相关但目前放在周期模块里的逻辑：

- `cycle.py::check_env_consistency()` 会比较 `温度` / `温度_内页`、`相对湿度` / `相对湿度_内页` 是否一致。

### 2.5 状态口径

建议状态口径：

- `PASS`：证书温湿度格式可解析，满足 KB 要求，且内外页一致。
- `FAIL`：温湿度明确超出 KB 要求，或内外页环境值明确不一致。
- `REVIEW`：证书温湿度缺失、KB 未命中、KB 字段不完整、LLM 无法判断。
- `ERROR`：温湿度向量库不可访问、collection 缺失、JSON 结构异常。

当前差异：

- 无有效温湿度库记录目前是“待定”，建议统一为 `REVIEW`。
- 内外页一致性目前在 `cycle.py`，建议后续移动或复用到环境模块，避免职责混杂。
- 当前只取区间第一个数字做 LLM 输入，后续应保留完整区间并由规则函数先判定。

## 3. 校准地点核验

### 3.1 业务需求

需求文档要求校准地点信息必须完整，能够支撑溯源路径清晰。地点应具体到以下层级之一：

- 建筑物，例如 `实验楼A座`
- 设施，例如 `恒温恒湿实验室`
- 房间名称或编号，例如 `203室`
- 楼栋、楼层、区域、车间、厂房、生产线等明确现场

如果校准地点为现场地点，且不是本机构授权实施的实验室，需要同步核查《认可的校准和测量能力表》中是否有 `*` 标识。

业务含义：

- `*` 表示该能力已通过 CNAS 现场校准能力专项评审。
- 现场校准缺少 `*` 标识时，应视为不符合要求。

### 3.2 输入

```json
{
  "仪器名称": "string",
  "校准依据": ["JJF/JJG/GJB ..."],
  "校准地点": "string",
  "委托方地址": "可选"
}
```

CNAS 能力库建议返回：

```json
{
  "仪器名称": "仪器名称，可含 *",
  "校准依据": "JJF/JJG/GJB ...",
  "distance": 0.0
}
```

地址库建议返回：

```json
{
  "校准地址": "标准地址",
  "专业室": "专业室",
  "序号": "序号",
  "distance": 0.0,
  "source": "来源"
}
```

### 3.3 如何比较

地点核验包含两条链路。

第一条链路：判断当前规程 / 仪器是否具备现场能力。

```text
证书校准依据
  -> CNAS 能力库同规程仪器
  -> 仪器名称是否带 *
```

第二条链路：判断证书地点是否命中地址库或足够具体。

```text
证书校准地点
  vs
地址库校准地址 / 向量检索 distance / 子串包含关系
```

当前代码按是否带 `*` 分流：

- 不带 `*`：地点必须命中地址库，或与地址库记录存在子串包含关系 / distance 小于阈值。
- 带 `*`：优先命中地址库；如果未命中，可由 LLM 或正则判断地点是否足够具体。

地点足够具体的判定线索包括：

- 房间号、门牌号、编号
- 楼栋、楼层、座、号楼
- 恒温恒湿实验室、屏蔽室、暗室、洁净室、计量室、校准室等设施
- 车间、厂房、生产线、区域、工位

### 3.4 当前实现

当前主代码位置：

- `langchain_app/checks/location.py`
- `langchain_app/retrieval/address.py`
- `langchain_app/retrieval/cnas.py`
- `langchain_app/graph/nodes/location_check.py`

当前实现行为：

- 从 JSON 读取 `仪器名称`、`校准依据`、`校准地点`。
- 通过 `search_instruments_by_basis_code()` 按规程号检索 CNAS 库，判断命中仪器名称是否带 `*`。
- 通过 `search_address_in_db()` 检索地址库。
- 地址命中方式包括：
  - 子串包含：地址库地址包含证书地点，或证书地点包含地址库地址。
  - 向量距离：`best_dist <= threshold`。
- 阈值来自配置：
  - `must_match_threshold`
  - `optional_match_threshold`
- 对带 `*` 的现场地点，未命中地址库时可走 LLM 或正则具体性判断。
- 对不带 `*` 的地点，未命中地址库时直接 FAIL。

### 3.5 状态口径

建议状态口径：

- `PASS`：不带 `*` 且地点命中地址库；或带 `*` 且地点命中地址库 / 地点描述足够具体。
- `FAIL`：校准地点为空；不带 `*` 且地点未命中授权地址库；带 `*` 但地点既未命中地址库又不够具体。
- `REVIEW`：CNAS 库或地址库返回结果不足，无法确认是否带 `*` 或无法确认地址语义。
- `ERROR`：`校准依据` 缺失、规程号无法解析、CNAS / 地址向量库访问异常。

当前差异：

- 当前 `check_location()` 在 JSON 中没有 `校准依据` 时直接返回错误文本。
- 当前 `search_instruments_by_basis_code()` 解析不到规程号会抛异常，外层没有统一转成结构化 `ERROR`。
- CNAS 库未检到仪器时，当前 `has_star` 默认为 `False`，可能导致现场能力未知时被当作“不带 *”处理；建议改为 REVIEW 或 ERROR 诊断。

## 4. 校准周期核验

### 4.1 业务需求

需求文档把日期选择分为三项：

- 接收日期
- 校准日期
- 建议校准周期

业务规则：

- 校准日期不能早于接收日期，只能等于或晚于接收日期。
- 大部分建议校准周期为 1 年。
- 建议校准周期应小于等于 1 年，或允许填写“无有效期”。
- 如果建议校准周期超过 1 年，必须核查校准规范、检定规程或仪器说明书是否明确支持该周期。
- 超出时间且无依据支持时，应直接异常处理。

### 4.2 输入

```json
{
  "委托单位": "string",
  "仪器名称": "string",
  "型号规格": "string",
  "校准依据": ["JJF/JJG/GJB ..."],
  "接收日期": "YYYY-MM-DD / YYYY/MM/DD / YYYY.MM.DD",
  "校准日期": "YYYY-MM-DD / YYYY/MM/DD / YYYY.MM.DD",
  "建议校准周期": "12个月 / 1年 / 无有效期"
}
```

周期库建议返回：

```json
{
  "仪器名称": "string",
  "依据": "依据名称或编号",
  "建议校准周期": "string",
  "来源": "通用数据库 / 华为数据库 / 默认标准",
  "distance": 0.0
}
```

### 4.3 如何比较

周期核验包含两类比较。

第一类是日期逻辑比较：

```text
接收日期 <= 校准日期
```

规则：

- 接收日期早于或等于校准日期，日期逻辑 PASS。
- 接收日期晚于校准日期，日期逻辑 FAIL。
- 任一日期缺失或格式无法解析，应 REVIEW 或 ERROR，当前代码标为 `[错误]`。

第二类是建议周期合理性比较：

```text
证书建议校准周期
  vs
客户专用周期库 / 通用周期库 / 默认周期
```

规则：

- 华为客户优先检索华为周期库。
- 华为周期库无匹配时回退通用周期库。
- 非华为客户直接检索通用周期库。
- 库中无匹配时可使用默认周期兜底。
- 证书周期应与命中的建议周期一致或不超过命中周期要求。
- 若证书周期超过 1 年，应检查知识库或依据文件是否明确允许。

### 4.4 当前实现

当前主代码位置：

- `langchain_app/checks/cycle.py`
- `langchain_app/retrieval/cycle.py`
- `langchain_app/graph/nodes/cycle_check.py`

当前实现行为：

- `check_date_logic()` 支持 `%Y-%m-%d`、`%Y/%m/%d`、`%Y.%m.%d` 三种日期格式。
- `check_date_logic()` 判断 `接收日期 <= 校准日期`。
- `check_cycle_reasonableness()` 读取 `委托单位`、`仪器名称`、`型号规格`、`校准依据`、`建议校准周期`。
- `委托单位` 包含 `华为` 时，优先检索华为周期库。
- 如果华为库未匹配，则检索通用周期库。
- 如果通用库仍未匹配，则使用 `cfg.default_cycle` 作为默认周期。
- 周期合理性由 LLM 输出 JSON：`find`、`reason`、`table`。
- 周期报告同时包含日期逻辑核验和温湿度内外页一致性核验。

### 4.5 状态口径

建议状态口径：

- `PASS`：日期逻辑正确，证书建议周期满足客户专用库 / 通用库 / 默认周期要求。
- `FAIL`：接收日期晚于校准日期；证书建议周期明确超过允许周期且无依据支持。
- `REVIEW`：建议周期字段缺失、周期库无匹配且只能使用默认周期、LLM 无法输出可靠判断。
- `ERROR`：周期向量库不可访问、JSON 结构异常、日期格式完全不可解析且无法人工判断。

当前差异：

- 当前无周期库匹配时会强制使用 `cfg.default_cycle`，并把 `find` 改成 1，容易掩盖知识库缺项；建议报告中显式标为 REVIEW 或至少标注“默认兜底”。
- 当前是否超过 1 年主要交给 LLM 和默认周期判断，建议增加规则函数先做确定性解析。
- 温湿度一致性目前放在周期报告中，建议后续移动到环境模块，周期模块只保留日期和周期。

## 5. 模块交付契约

### 5.1 PDF 解析组需要交付

JSON 字段至少包括：

- `温度`
- `相对湿度`
- `温度_内页`
- `相对湿度_内页`
- `校准地点`
- `委托方地址`
- `接收日期`
- `校准日期`
- `建议校准周期`
- `仪器名称`
- `型号规格`
- `委托单位`
- `校准依据`

字段缺失时应保留空值，并尽量保留原始文本，不能把无法识别的值猜成合格值。

### 5.2 RAG 组需要交付

温湿度库：

- `仪器名称`
- `FILE_CODE`
- `FILE_NAME`
- `温度要求`
- `相对湿度要求`
- `最大温度变化范围`
- `认可组织`
- `source`
- `distance`

地址库：

- `校准地址`
- `专业室`
- `序号`
- `source`
- `distance`

周期库：

- `仪器名称`
- `依据`
- `建议校准周期`
- `来源`
- `source`
- `distance`

CNAS 库：

- `仪器名称`
- `校准依据`
- 是否带 `*` 的原始仪器名称
- `distance`

### 5.3 核验组需要交付

每个模块都应输出可解释 Markdown，并在内部保留结构化结果：

```json
{
  "module": "environment/location/cycle",
  "status": "PASS/FAIL/REVIEW/ERROR",
  "evidence": [],
  "reason": "判定说明",
  "diagnostic": {
    "code": "OK/KB_MISSING/DB_ERROR/PARSE_ERROR/LLM_ERROR",
    "message": "诊断信息"
  }
}
```

验收重点：

- 能说明比较对象是谁。
- 能说明为什么 PASS / FAIL / REVIEW / ERROR。
- 知识库缺项和证书不合格必须区分。
- LLM 不可用不能直接导致业务 FAIL，应转为 REVIEW 或 ERROR。
- 能保留检索来源、distance、默认兜底等证据。
